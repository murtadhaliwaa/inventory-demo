# مخالفات الممارسات الحالية وديون تقنية (مرجع للتحسين)

> الهدف: **عدم إخفاء** نقاط الضعف؛ التطوير الجديد يُفضَّل ألا يكررها. للهيكلية المثلى عند الإضافة راجع مجلد `الهيكلية-الصحيحة-والمرجعية/`.

---

## 1) ملف إجراءات واحد كبير (`src/lib/actions/inventory.ts`)

**الوصف:** معظم منطق WMS في ملف واحد (مواد، موردين، حركات، تقارير، PDF، ترقيم صفحات).

**لماذا تعتبر دينًا:** صعوبة المراجعة، احتمال تعارض دمج Git، اختبارات أصعب.

**التحسين المقترح:** تقسيم إلى `lib/actions/inventory/*.ts` حسب النطاق (انظر `02-دليل-إضافة-خاصية-جديدة.md`).

---

## 2) تكرار نمط النماذج في الواجهة (Form + Transition + Toast)

**المواقع النموذجية:**

- `item-form-dialogs.tsx`
- `daily-operations-forms.tsx`
- `suppliers-table.tsx` (نموذج المورد)

**الوصف:** نفس التسلسل: `preventDefault` → `safeParse` → `useTransition` → استدعاء action → `toast` → `router.refresh` أحيانًا.

**التحسين المقترح:** Hook مشترك مثل `useServerActionForm(action)` أو مكوّن `<ServerActionForm>` يقلل التكرار — **بعد** توحيد شكل نتيجة الـ actions (`{ success, error }`).

---

## 3) قائمة `revalidatePath` يدوية (`paths` في `inventory.ts`)

**الوصف:** أي مسار جديد قد يُنسى فيُعرض محتوى قديم حتى يحدث تحديث يدوي.

**التحسين المقترح:** دالة `revalidateInventory()` تجمع المسارات، أو تقسيم revalidate حسب النطاق مع تصدير من كل وحدة.

---

## 4) استعلامات خام (`$queryRaw`) مختلطة مع Prisma ORM

**المواقع:** تنبيه النقص في `getDashboardData` / `getDailyReport` / `getLowStockItems`.

**السبب:** قيود Prisma على مقارنة عمودين.

**التحسين المقترح:** الإبقاء مع تعليق قصير + اختبار فهرسة؛ أو View في SQL + نموذج Prisma `@@ignore` للقراءة فقط (متقدم).

---

## 5) حد أقصى لقوائم العمليات (`getOperationsPageData` → `OPS_SELECT_LIMIT`)

**الوصف:** لمنع بطء القوائم المنسدلة، يُجلب حتى 3000 صف لكل من المواد والموردين.

**المخاطر:** إذا تجاوزت المنظمة هذا العدد، لن تظهر كل الخيارات.

**التحسين المقترح:** بحث خادمي (combobox async) أو ترقيم + بحث نصي.

---

## 6) صلاحيات الحذف مبنية على البريد فقط (`WMS_ADMIN_EMAILS`)

**الوصف:** ليست RBAC كاملة (أدوار متعددة لكل عملية).

**التحسين المقترح:** أدوار في `app_metadata` في Supabase + طبقة `assertRole()` لكل عملية.

---

## 7) تكرار محتمل في عرض الجداول

**الوصف:** بعض الجداول تستخدم `overflow-x-auto` + `wms-panel`، وأخرى تستخدم `Card` + `border` فقط.

**التحسين المقترح:** مكوّن `<DataTableShell>` واحد للهوامش والتمرير.

---

## 8) PDF على العميل (`html2canvas` + `jsPDF`)

**الوصف:** ثقيل على المتصفح مع تقارير ضخمة؛ منطق التصدير في عميل.

**التحسين المقترح (طويل المدى):** توليد PDF على الخادم (مثلاً `@react-pdf` أو قالب HTML → PDF) لحدود أحجام أكبر وأمان أعلى.

---

## 9) عدم وجود اختبارات آلية موثقة

**الوصف:** الاعتماد على `npm run build` + فحص يدوي.

**التحسين المقترح:** إضافة Vitest/Playwright تدريجيًا لمسارات حرجة (تسجيل دخول، إضافة حركة، منع حذف مادة لها حركات).

---

## 10) توثيق الـ API الخارجي

**الوصف:** النظام حاليًا تطبيق ويب متكامل وليس REST عامًا.

**ملاحظة:** إن أضيفت REST لاحقًا، يجب **عدم** تكرار منطق الأعمال؛ استدعِ نفس الخدمات الداخلية التي تستخدمها Server Actions.
